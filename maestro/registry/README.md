# Registry

The registry is a set of auto-generated Python modules that provide strongly-typed access to Home Assistant entities and their attributes. Modules are generated by `RegistryManager` into the user project's `registry_dir` (configured on `MaestroApp`, default `registry/`) and imported by user code as a top-level package (e.g. `from registry.light import desk_lamp`).

## How It Works

- **Programmatic Updates**: Each module is generated and updated by `RegistryManager` -- edit entity entries with caution
- **Module Parsing**: Line-by-line formatting is critical for parsing and editing registry modules; do not alter the format
- **Pruning**: `RegistryManager.prune()` removes entries for entities no longer present in Home Assistant. It raises if more than 25% of the registry would be removed, unless called with `force=True`
- **Disaster Recovery**: Deleting a registry entity (or entire module) and allowing it to recreate should fix most problems
- **Ownership**: Generated modules live in the user's project; whether they are committed or gitignored is the user's call

## Generated Module Structure

Each generated module contains one class per entity (subclassing the appropriate domain class) with `EntityAttribute` descriptors, plus a module-level instance:

```python
# registry/light.py (auto-generated)

from maestro.domains import Light
from maestro.domains import EntityAttribute
from datetime import datetime

class LightMultiColorBulb(Light):
    brightness = EntityAttribute(int)
    rgb_color = EntityAttribute(list)
    color_mode = EntityAttribute(str)
multi_color_bulb = LightMultiColorBulb("light.multi_color_bulb")
```

## Custom Domain Subclasses

Registry parent classes can be swapped for custom domain subclasses to extend functionality. Edit the generated class's parent (e.g. `class ClimateThermostat(Thermostat):`); `RegistryManager` preserves the custom parent through updates and prunes, provided the module format is kept intact.

The generator resolves each parent class to its defining package and emits imports accordingly -- built-in parents from `maestro.domains`, custom parents from the user's custom domains package (configured via `custom_domains_dir` on `MaestroApp`):

```python
# registry/climate.py (auto-generated)

from maestro.domains import Climate
from custom_domains import Thermostat

class ClimateThermostat(Thermostat): ...
thermostat = ClimateThermostat("climate.thermostat")
```

Custom domain classes must be defined in the configured custom domains package and exported via `__all__` in its `__init__.py`. Maestro imports the package during `MaestroApp` construction, before registry modules and scripts load.
